# Code Audit — 2026-07-07

Independent review of the Insights Engine (Steps 11–13). Two audits were merged: a primary review and an independent second pass.

---

## Findings

| ID | Area | Severity | Finding |
|----|------|----------|---------|
| F1 | SalesSummary write path | High | `deplete_batch` did not write to `sales_summaries`; `get_covers_insight` therefore always returned null `avg_check`. Additionally, `labor_service` used `SalesSummary` for DOW bucketing — no rows → zero revenue everywhere. |
| F2 | Daypart / DOW timezone | High | `get_sales_patterns` bucketed by UTC weekday and UTC hour. A Friday 11pm sale in New York landed in Saturday's bucket; dinner-hour sales appeared as late-night. |
| F3 | Channel-aware depletion | High | `deplete_from_sale` applied all recipe lines unconditionally regardless of `RecipeLine.channel`. Delivery-only packaging was consumed for dine-in sales. |
| F4 | Tab deep links | High | `InsightsPage` tab state was local (`useState` only). Sharing a URL or reloading always landed on the default tab, not the intended one. |
| F5 | Query invalidation after sales | Medium | `QuickSalesEntry` did not invalidate `food-cost`, `profitability`, `insights`, or `alerts` query caches on success. Dashboard showed stale data until the next automatic refetch. |
| F6 | Settings flush race | Medium | `get_or_create_settings` could raise `IntegrityError` on concurrent first requests (unique constraint on `restaurant_id`). The session was left in an error state. |
| F7 | Action service silent failures | High | All six `except Exception: pass` blocks in `get_daily_actions` silently swallowed errors. A broken sub-service caused the entire action list to return incomplete without any log entry. |
| F8 | Missing hot-table indexes | Medium | `sale_adjustments` and `labor_entries` had no index on `(restaurant_id, business_date)`. Full-table scans on every insights request. |
| F9 | Free-form string validation | Low | `channel` on `SaleItem`/`ChannelFeeCreate` and `adjustment_type` on `AdjustmentCreate` were unvalidated strings. Typos (`'doordash'`, `'Comp'`) silently produced orphan data. |
| F10 | Verdict string duplication | Low | `action_service` duplicated the verbose verdict string as a bare literal. Renaming the verdict in `price_experiment_service` would silently break the action filter. |
| F11 | Misc cleanups (5 sub-items) | Low | (a) `get_contribution_margins` missing tenant guard; (b) `get_price_experiments` silently skipped deleted items instead of emitting a placeholder; (c) `actionsData` query had no `staleTime`, refetching on every tab focus; (d) `SECRET_KEY` dev placeholder not warned on startup; (e) redundant null guard before `< -15` numeric compare. |
| F12 | Documentation | — | DECISIONS.md missing entries for upsert strategy, timezone bucketing, recipe channel semantics, and Literal validation. Audit adjudication section required. |
| COR-01 | `_window_metrics` date comparison | Medium | Alleged: compared `TIMESTAMPTZ` values against a `DATE` column; PostgreSQL cast caused incorrect windowing. **FALSE PREMISE** — `SalesByItem.business_date` is `DateTime(timezone=True)` (TIMESTAMPTZ), not DATE. Original aware-datetime comparison was exact and correct. See adjudication. |
| N+1-01 | N+1 in `get_price_experiments` | Medium | Each event issued a separate `db.get(MenuItem, ...)` and two sub-queries. **Explicitly deferred** — not a correctness issue; address with a JOIN rewrite when query counts are measured in production. |
| N+1-02 | N+1 in `get_contribution_margins` | Low | Per-item `db.get(Ingredient, ...)` calls inside a loop. **Explicitly deferred** — same reason as N+1-01. |

### Verified false positives

| ID | Verdict | Evidence |
|----|---------|----------|
| SEC-03 | **FALSE POSITIVE** | Alleged: PATCH `/menu-items/{id}` missing ownership check. `routers/recipes.py` fetches the item and raises 404 when `mi.restaurant_id != rid` before any mutation. |
| COR-01 | **FALSE PREMISE — REVERTED** | Alleged: `_window_metrics` compared TIMESTAMPTZ against a DATE column. `SalesByItem.business_date` is `DateTime(timezone=True)` (TIMESTAMPTZ), not DATE. The original `>= start, < end` aware-datetime comparison was exact and correct. The `.date()` truncation applied in `fce00c4` made windows coarser (day-of-change sales misattributed to the after window; denominator day-count still used exact timestamps). Reverted in the follow-up commit. |
| TXN-02 | **FALSE POSITIVE** | Alleged: benchmark computation non-atomic. `run_benchmark_computation` issues a single `db.commit()` at the end of each metric's write. The only intra-loop commit was `get_or_create_settings`, resolved by F6 (flush-only, caller commits). |
| TXN-03 | **FALSE POSITIVE** | Alleged: `confirm` endpoint not idempotent. `ingestion_commit_service` raises when `staged.status != 'pending'`; a double-confirm returns an error on the second call rather than double-applying. |
| DEAD-01 | **FALSE POSITIVE** | Alleged: dead LightGBM stubs still in codebase. No `lightgbm` references exist in any service file; the import was removed when forecasting was gated in Step 8. |
| DEAD-03 | **FALSE POSITIVE** | Alleged: weather fetch not scheduled. `fetch-nightly-weather` is in the Celery `beat_schedule` at 4:00 UTC in `app/celery_app.py`. |
| PERF-06 | **FALSE POSITIVE** | Alleged: partition indexes dropped by the Step 11/12 migration. The migration strips only `drop_index` lines produced by TimescaleDB autogenerate artifacts; it does not drop any index that was intentionally created. The actual indexes were never removed. |
| SEC-01 | **DOWNGRADED TO NIT** | Alleged: cross-tenant `RecipeLine` leak — `RecipeLine` has no `restaurant_id` column so the proposed guard would crash. In practice, ingredient UUIDs are globally unique and tenant-scoped via `MenuItem → restaurant_id`, so a cross-tenant RecipeLine cannot match a legitimate query. Optional defense-in-depth join (MenuItem → restaurant check) deferred. |
| FE-05 | **CORRECTED** | Alleged: alerts poll interval 30s with background refetch. Actual: `refetchInterval: 60_000` (60s). `refetchIntervalInBackground` defaults to `false` in TanStack Query v5; background tabs do not poll. |

### Deferred

| ID | Description |
|----|-------------|
| PERF-01 | N+1 in `get_price_experiments` — per-event `db.get(MenuItem)` + two sub-queries. Rewrite with JOIN when production query counts are measured. |
| PERF-02 | N+1 in `get_contribution_margins` — per-item `db.get(Ingredient)` in loop. Same rationale. |
| PERF-03 | N+1 in `get_variance_report`, `get_par_recommendations`, `get_sensitivity_analysis`, `run_benchmark_computation` — various per-row sub-queries. Batch with `IN` clause or subquery join. |
| SEC-02 | JWT stored in `localStorage` — vulnerable to XSS in a browser context. Acceptable for a local-network operator tool pre-deployment; migrate to `httpOnly` cookies before any internet-facing deployment. |
| FEAT-01 | Per-platform delivery channel granularity — `channel='delivery'` groups DoorDash, Uber Eats, etc. into one bucket. Add platform-level channel strings when delivery integrations are built. |
| QUAL-01 | `as any` casts in `InsightsPage.tsx` and `Dashboard.tsx`. Clean up with proper response types once the API response shapes stabilise. |
| QUAL-02 | `window_days` not included in TanStack Query keys — stale data served if the user changes the window. Add `window_days` to the `queryKey` arrays. |
| TXN-04 | `ingestion_commit_service` confirm path has no `SELECT FOR UPDATE` on the staged row. Concurrent confirms on the same row could double-apply in theory. Low risk given single-operator usage; add advisory lock or `FOR UPDATE` before any multi-user deployment. |
