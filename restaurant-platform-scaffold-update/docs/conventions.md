# Conventions

Read this before writing any code. These are the naming, structure, and pattern rules that keep the codebase consistent across many build sessions.

---

## Directory layout

```
restaurant-platform/
├── backend/
│   ├── app/
│   │   ├── main.py            # FastAPI entry point, router registration
│   │   ├── config.py          # pydantic-settings, reads .env
│   │   ├── database.py        # engine, SessionLocal, get_db, Base
│   │   ├── models/            # SQLAlchemy ORM models (one file per domain)
│   │   ├── schemas/           # Pydantic request/response models
│   │   ├── routers/           # HTTP handlers — thin, delegate to services
│   │   ├── services/          # ALL business logic lives here
│   │   └── workers/           # Celery app + tasks
│   ├── alembic/
│   ├── requirements.txt
│   └── .env                   # never commit
├── frontend/
│   └── src/{components,pages,hooks,api}/
└── docker-compose.yml
```

---

## Layer responsibilities (strict)

- **models/** — data shape only. SQLAlchemy columns, relationships, and `@property` computed fields (like `food_cost_pct`). No business logic.
- **schemas/** — Pydantic models for API request/response validation. Separate from ORM models.
- **routers/** — thin HTTP layer. Parse request, call a service, return response. No business logic in routers.
- **services/** — all business logic. Depletion, food cost math, ingestion, alerts, insights. Services take a `db: Session` and return data or ORM objects.
- **workers/** — Celery tasks that call services on a schedule.

If you're writing business logic in a router, stop — it belongs in a service.

---

## Naming

- **Files:** snake_case (`food_cost_service.py`, `insights_service.py`).
- **Models:** PascalCase singular (`MenuItem`, `VendorInvoice`, `StagedIngestion`, `LaborEntry`).
- **Tables:** snake_case plural (`menu_items`, `vendor_invoices`, `labor_entries`).
- **Service functions:** verb_noun (`process_invoice`, `deplete_from_sale`, `get_variance_report`).
- **Routers:** prefix matches domain (`/menu-items`, `/sales`, `/ingestion`, `/insights`).
- **React components:** PascalCase (`KpiCard.tsx`, `ReviewIngestion.tsx`).
- **React pages:** PascalCase in `pages/` (`Dashboard.tsx`, `InsightsPage.tsx`).

---

## Database conventions

- **Primary keys:** UUID, `default=uuid.uuid4`.
- **`restaurant_id`** foreign key on every restaurant-scoped table. Non-negotiable.
- **Timestamps:** `created_at` with `server_default=func.now()`. Add `updated_at` with `onupdate=func.now()` where records change.
- **Money:** `Numeric(10,2)` for totals, `Numeric(10,4)` for per-unit costs (4 decimal places matters for ingredient cost precision).
- **Time-series tables** (`sales_summaries`, `sales_by_item`): TimescaleDB hypertables, partitioned on `business_date` — composite PK `(id, business_date)` is required by Timescale.
- **Never overwrite point-in-time data.** Inventory counts, sales, invoice prices are append-only timestamped records.
- **Review autogenerated Alembic migrations** and strip spurious `drop_index` lines on hypertable partition indexes before `alembic upgrade head`.
- **Normalize IDs with `_to_uuid()`** before comparisons — strings from URLs/JSON ≠ UUID objects.

---

## API conventions

- **Auth:** every data router injects `rid: str = Depends(get_current_restaurant_id)`.
- **Scoping:** every query filters by `restaurant_id == rid`. Verify ownership on single-record fetches (`if obj.restaurant_id != rid: raise HTTPException(404)`).
- **Status codes:** 201 for creates, 404 for not-found/wrong-tenant, 401 handled by auth dependency.
- **Partial updates (PATCH):** use Pydantic `model_fields_set` to update only fields the client actually sent.
- **Test via `/docs`** (FastAPI auto-generated) before building frontend for an endpoint.

---

## Ingestion conventions (critical)

- Every automated path (CSV, OCR, voice, email) writes to `staged_ingestions` first — never straight to live tables.
- The operator review UI confirms or rejects.
- On confirm, `ingestion_commit_service.commit_staged_ingestion` translates staged data to live records using the **same** services as manual entry (`process_invoice`, `deplete_batch`).
- Fuzzy-match extracted ingredient names against the catalog (`difflib.get_close_matches`, cutoff 0.7). Auto-create the ingredient if no match.
- Confidence scores drive UI highlighting: green > 0.85, yellow 0.6–0.85, red < 0.6.
- `business_date` from date-pickers: construct at **noon UTC** to avoid timezone off-by-one.

## Workflow per step

1. `git commit` the previous step before starting a new one (recoverability).
2. Read the step file + this conventions file.
3. Summarize what you'll build before writing code.
4. Build it. Run `alembic upgrade head` if the schema changed.
5. Verify with a throwaway script (seed → exercise → assert → clean up seed data → delete script). Frontend: `npm run build` must report 0 TS errors, then visual check.
6. Update the checkbox in `CLAUDE.md`.
7. Log any non-obvious choice in `DECISIONS.md`.
8. `git push` when the step is verified.

## Insights layer rules

- **All insight math lives in `services/insights_service.py`** (split into `insights_*_service.py` files if any single file exceeds ~500 lines). Routers under `routers/insights.py`, prefix `/insights`.
- **Insights are derived, never stored** (same philosophy as food cost): compute from live data on request. Exception: `benchmark_stats` (Step 13) is a nightly *aggregation* table because it's cross-tenant — the one deliberate precompute.
- **Every insight response includes its inputs**: return the underlying numbers (numerator/denominator, window, thresholds used) alongside the headline figure, so the frontend can always show "how was this computed."
- **Every insight ends in an action**: response schemas include a `recommended_action: str | None` field. If there's nothing to do, it's `None` — never invent busywork.
- **Windows are parameters with defaults**, not hardcoded: `window_days: int = 28` style. Defaults: variance 7d, inflation 30/60/90d, velocity 28d, patterns 28d.
- **Thresholds live in `restaurant_settings`** with sane defaults (menu-engineering popularity factor 0.7, par cover bounds 4/21 days, etc.) so operators can tune without code changes.
- **Cross-tenant aggregation (Step 13 only):** minimum-cohort rule — never return a benchmark computed from fewer than 5 restaurants; aggregate nightly via Celery; store only anonymized aggregates (metric, cohort, percentiles), never per-restaurant values in the shared table.
- **Divide-by-zero discipline:** insight math is full of ratios (margins, percentages, velocities). Guard every denominator; return `None` + a `data_gap: str` explanation rather than 0 or a crash ("no closing count in window — variance unavailable").
- **Data-coverage honesty:** when an insight's inputs are partial (e.g., daypart patterns but only 60% of sales carry timestamps), include `coverage_pct` in the response and render it in the UI. Never present partial data as complete.
- **Frontend:** one `InsightsPage.tsx` with cards/tabs per insight; the daily action list (Step 13) renders as the FIRST card on `Dashboard.tsx`. Reuse `KpiCard`; charts via Recharts; all fetching via TanStack Query with `invalidateQueries` on any mutation that changes underlying data.
